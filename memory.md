## Dataset lessons — FAERS

### Source / pipeline
- Source: openFDA `/drug/event.json` (serious events only, primary filter).
- Pipeline: `pipeline/pipelines/faers_weekly.py` — Mondays 03:00 UTC via `faers-weekly.yml`. Adaptive weekly→daily chunking with >25k overflow detection. Per-table hard-coded upsert chunk sizes: `PARENT_CHUNK_SIZE=100` for `drug_adverse_events` (JSONB payload), `CHILD_CHUNK_SIZE=500` for drugs/reactions, `DIM_CHUNK_SIZE=500` for dim tables.
- Volume: 1,583,293 serious events, 8.16M drug rows, 5.57M reaction rows. Spans Jan 2024 – March 2026 (27 months). openFDA refresh lags 6–12 weeks; April 2026+ not yet published as of May 21. Weekly cron catches new months automatically.
- Consumer baseline: 36.61% across full dataset. Reference for per-drug lawyer-flood proxy.

### Data shape lessons (vs predictions)
- 2026-05-15 — **Drug match-path distribution is 87.7% name-fallback / 12.3% NDC, NOT NDC-heavy as predicted.** FAERS reports often lack NDCs, and when NDC fails the chain falls all the way through (UNII/RxCUI/AppNo only trigger when NDC enrichment succeeds first).
- 2026-05-15 — **`primarysource_qualification=4` (lawyer) is only 0.45% of reports, NOT 5–15% as predicted.** Mass-tort intake routes through manufacturers, inflating consumer (5 = 35.6%) not lawyer. Real lawyer-flood detection needs anomaly analysis (consumer-rate spike vs baseline) or `companynumb` pattern matching, not a simple qualification filter.
- 2026-05-15 — **`drug_manufacturer_aliases` is empty by design** (PR #382 — pipeline records misses, table is human-curated). Will need seeding for named torts in PR-4.
- 2026-05-21 — **FAERS canonical preferred terms are NOT what their common names suggest.** Gastroparesis MDL injury is filed under "Impaired gastric emptying" (3,342 occurrences) — literal "Gastroparesis" appears zero times. NAION (MDL 3163 core injury) is filed under "Optic ischaemic neuropathy" (767 occurrences) — literal "NAION" appears zero times. Lesson: never assume FAERS uses the layperson injury name; always investigate canonical PTs against actual data before building term filters.

### Query / index gotchas
- 2026-05-21 — **Substring ILIKE over the 8.16M-row `drug_adverse_event_drugs.medicinalproduct` table times out at the 8s Postgres query limit.** Exact-match arrays against the existing btree index (`medicinalproduct = ANY(ARRAY[...])`) are the only viable matching strategy without adding trigram indexes.

### Per-tort canonical terms (verified against live data)
- 2026-05-22 — **Depo-Provera:** brand `["DEPO-PROVERA"]` (oral PROVERA + generic MPA excluded); injury = meningioma, literal `Meningioma` PT exists (8-term spectrum). 95.2% lawyer-sourced → lawyer concentration mode. See `faers-depo-provera.ts`.
- 2026-05-22 — **Dupixent (dupilumab) has NO single dominant injury in FAERS.** Largest reaction clusters are ocular-surface disease (~1,608 reports) and joint/arthritis (~1,604) — but both are **0.0% lawyer-filed** (on-label ADRs, no litigation footprint). The litigation-shaped cluster is **cutaneous T-cell lymphoma**: 198 qualifying reports, **5.56% lawyer (7.6× the 0.73% baseline)**, 4.5% fatal — theory is dupilumab unmasking CTCL misdiagnosed as atopic dermatitis. Chosen as the Dupixent tort. Brand map `["DUPIXENT","DUPILUMAB"]` — generic INN **included** here (departure from GLP-1/Depo rule) because dupilumab is a single-source biologic with no generic competitor, so the INN maps 1:1 to the brand. CTCL PTs: FAERS codes these as `Cutaneous T-cell lymphoma`, NOT `Mycosis fungoides`/`Sezary syndrome` (neither PT appears). Bare `Lymphoma` excluded — verified 0.0% lawyer, B-cell-ambiguous. See `faers-dupixent.ts`.

### Core finding: two streams, different sources
- **Incident news** (crash happened, worker died, suit filed) — high volume, fresh daily, easy to auto-feed.
- **Outcome news** (verdict/settlement $X) — lower volume, higher value, harder to source. This is the differentiator; an incident-only banner is commodity a competitor can clone.

### Hard constraint: republish wall (drives the whole design)
- 2026-06-25 — **No paid verdict database can be republished in the SaaS.** ALM/VerdictSearch terms flatly prohibit reproduction/redistribution without written consent; Law360/LexisNexis same; Alabama Jury Verdict Reporter (AJVR) is a small copyrighted publication. Underlying verdict *facts* aren't copyrightable; their written *case reports* are.

### Approved source stack (republishable, national-scalable)
1. **Government data (public domain — display facts directly):**
   - **OSHA press releases** — covers construction/explosion/electrocution/trench/confined-space. URL filter works: `https://www.osha.gov/news/newsreleases?search_state=<State>&year=<YYYY>`. Structured: date, headline, city, employer, hazard, fatality/injury count, stable permalink. AL volume ~1–3/mo.    - **CourtListener (MCP, free)** — federal filings, all 94 districts. Query spec: `type=d`, `court=["alnd","almd","alsd"]`, fields `caseName,dateFiled,docketNumber,court,cause,juryDemand,firm`, `filed_after` rolling, `order_by=dateFiled desc`. **Big limitation confirmed by live test:** AL single-incident PI is overwhelmingly STATE circuit court (Jefferson/Mobile/Madison), which CourtListener/PACER does NOT cover. Trucking returned 1 federal case in a year; MVA query was dominated by 3M-earplug removals + contract disputes. CourtListener = clean-metadata garnish, NOT the backbone.
2. **News headlines via Google News (display headline + source + date + link, aggregator-style — NOT full article text):**
   - **SearchApi.io** (Lance already has it) — `google_news` engine, real-time, low-latency, pay-per-search. Primary live per-state fetcher. NOTE: the connected SearchApi MCP only exposed `google_search_light` (web), not a news endpoint — the `google_news` engine is called via SearchApi's HTTP API directly.
   - **Apify** (Lance already has it) — Google News actors, batch/scheduled, pay-per-article. ## Feature research — Strategy Engine (PI media-strategy recommendations)

Research-only as of 2026-06-25 (Cowork ideation, no code). Sibling to the State Legal News banner — both are state-page features, Alabama-first then national, PI-specific (mass-tort version mirrors later, tort-specific). Pre-council framing; Lance plans to run the model-council skill to pressure-test before building.

### The problem it solves
State pages surface unconventional data but stop at "here's a data point." Build Campaign produces creative assets but with no strategy to hang them on. The Strategy Engine is the missing middle layer: read the surfaced market data + a few positioning inputs → output 2-3 reasoned media strategies the user can read as cards and export to PDF/slides. Must serve all three personas (firm plans its own entry; agency white-labels for a client; media seller justifies inventory) from one engine, different reader.

### Proposed architecture: rules-and-data scaffold, AI as writer NOT strategist
Do NOT have the LLM invent media theory per request (produces plausible mush, hallucinates, not defensible, clonable). Instead encode the expert decision framework as structure; let data select/justify; let AI personalize + write. Four moving parts:
1. **Goal + posture → strategy archetype** (fixed set, ~5): Head-to-Head (outspend leader on same channels), Niche Carve-Out (own a county/case-type the big players ignore), Audience Play (own an under-served demographic — the Greene County Urban-radio example), Always-On Brand (sustained awareness, established firm), Surge (event-driven, e.g. post-mass-casualty). Engine picks the 2-3 that fit.
2. **Market data → opportunity:** the existing Competitive Analysis data (paid-search/SEO/YouTube/Meta firm rosters per DMA) selects + justifies the archetype. Whitespace = if 23 firms saturate paid search but nobody runs Urban radio, the Audience Play archetype lights up. Data isn't decoration — it's the selector.
3. **Audience → channel → funnel:** demographic baseline + format-genre CSV map target audience to channels/outlets, sequenced by funnel stage (awareness = broadcast radio/TV/cable wide reach by format; consideration = programmatic/YouTube/Meta behavioral; conversion = GroundTruth geofencing of accident-adjacent locations + paid search high-intent + retargeting). Engine must explain WHY this channel at this stage for this audience.
4. **Budget → allocation:** rough CPM/cost ranges split dollars across channels. Even crude makes it feel real. (Repo has `cost-benchmarks` + `cpa-estimate` surfaces — candidate source; v1 could stay qualitative.)
**The moat = every recommendation traces to a data point visible on the page** ("run Urban AC in Birmingham DMA because (a) target over-indexes per Pew, (b) WBHK/WBHJ cover Greene County, (c) zero competitors buying it — proof in your own competitive panel"). Not generatable without LMI's data.

### Output + intake (Lance's leanings, not locked)
- **Output:** archetype cards on the page → user confirms a couple inputs → generate full strategy as **downloadable PDF or slides** (channel-by-funnel plan, named outlets, audience rationale, budget split). Lance liked cards + "answer questions then download."
